# Eurekan.org 프로젝트 실행 지식 베이스

## 문서 목적
- 이 문서는 Eurekan.org 프로젝트의 최상위 실행 규칙이다.
- 이 문서의 모든 내용은 반드시 한글로 유지한다.
- 현재 저장소의 진실은 실제 파일 구조와 실제 코드다.
- 코드가 바뀌면 같은 변경에서 이 문서도 함께 갱신한다.

## 현재 프로젝트 정체성
- 프로젝트명: Eurekan.org
- 제품 형태: 정적 단일 페이지형 위키/매거진
- 핵심 목표: 지금 사람들이 가장 궁금해하는 주제를 분야별 장문 페이지로 보여주는 읽기 중심 사이트
- 현재 구현 원칙: 백엔드 없이 `README.md`, `index.html`, `style.css`, `app.js`, `data.json`, `robots.txt`, `sitemap.xml`, `pages/`, `agents/`, `images/`만으로 동작한다.

## 현재 파일 구조
```text
AGENTS.md
README.md
index.html
style.css
app.js
data.json
robots.txt
sitemap.xml
images/
  .gitkeep
pages/
  <slug>.md
  <slug>.html
agents/
  README.md
  <desk>-agent-<name>.md
```

- 위 구조 외의 프로젝트 파일/폴더는 유지하지 않는다.
- `README.md`는 공개 저장소용 프로젝트 소개 문서로 유지한다.
- `agents/` 폴더는 분야별 전문가 페르소나 문서 저장용으로 남긴다.
- `images/` 폴더는 생성형 이미지 또는 직접 관리하는 공개용 이미지 자산 저장용으로 남긴다.
- 실제 사용자-facing 사이트는 루트의 `index.html`, `style.css`, `app.js`와 `pages/<slug>.html` 정적 글 페이지를 사용한다.
- 검색엔진 수집 보조 파일은 루트의 `robots.txt`, `sitemap.xml`만 사용한다.
- 정적 파일 경로는 배포 위치에 덜 민감하도록 상대 경로를 우선 사용한다.

## 진실의 원천 우선순위
1. 실제 루트 파일 구조와 실제 코드
2. 이 문서 `AGENTS.md`
3. `pages/*.md` 본문 원본과 `pages/*.html` 생성 결과
4. `agents/<desk>-agent-<name>.md` 분야별 전문가 참고 문서

## 전역 규칙
- 모든 신규 문서형 운영 지침은 한글로 작성한다.
- `README.md`의 설명 문구와 공개 안내 문구는 한글 존댓말로 작성한다.
- OpenCode에서 사용자에게 설명하거나 안내하는 문구는 한글 존댓말로 작성한다.
- 사용자에게 보이는 공개 문구와 글 본문은 `~입니다`, `~합니다`, `~습니다` 중심의 존댓말을 사용하고, `~이다`, `~다`로 끝나는 반말·평서체 문장을 쓰지 않는다.
- 글 제목과 소제목은 존댓말 의무 대상에서 제외하며, 독자가 빠르게 구조를 파악할 수 있는 짧은 질문형·명사형 문구를 사용할 수 있다.
- 모든 신규 git commit 메시지는 한글로 작성한다.
- 과도한 추상화 없이 단순하게 유지한다.
- Python 백엔드, 관리자 페이지, 승인 API, 데이터베이스, 서버 라우터를 다시 도입하지 않는다.
- 페이지 렌더링과 상호작용은 `app.js` 하나에서 처리한다.
- 스타일은 `style.css` 하나에서 처리한다.
- 페이지 본문 원본은 `pages/<slug>.md` 파일로 저장하고, 검색엔진용 정적 글 페이지는 같은 slug의 `pages/<slug>.html` 파일로 생성한다.
- 페이지 메타데이터 목록은 `app.js` 내부 manifest 배열에서 관리한다.

## 금지 사항

- Python/Node 백엔드 재도입 금지
- 관리자 승인 플로우 재도입 금지
- 로그인/회원가입/결제 기능 도입 금지
- 페이지 외부에 별도 콘텐츠 저장 포맷 추가 금지

## 사이트 동작 원칙
- 헤더는 스크롤해도 상단에 고정하지 않고 일반 문서 흐름에 둔다.
- 헤더 안에는 가운데 정렬된 `💡 EUREKAN` 브랜드 제목과 분야별 메뉴만 둔다.
- 헤더 안의 브랜드 제목과 분야별 메뉴는 모두 가운데 정렬한다.
- 분야별 메뉴는 헤더 안에서 브랜드 제목 아래에 두고, 항상 1줄 가로 메뉴로 표시한다.
- 분야별 메뉴가 화면 폭보다 길어지면 좌우 가로 스크롤로 탐색하게 하되, 메뉴 영역의 스크롤바는 시각적으로 숨긴다.
- 화면 폭이 520px 이하로 매우 좁아지면 분야별 메뉴 버튼을 숨기고 제목 줄 오른쪽에 우측 정렬된 햄버거 버튼으로 대체하며, 햄버거 버튼을 누르면 분야별 메뉴가 흰색 드롭다운 패널로 펼쳐진다.
- 검색창은 헤더 안에 넣지 않고 메인 영역의 최상단에 두며, 헤더와 콘텐츠 사이 간격이 과하게 벌어지지 않도록 상하 여백을 작게 유지한다.
- 사이트 최상위 배경과 헤더 배경은 흰색 계열로 통일하고, 화면 구분은 배경색 차이가 아니라 얇은 경계선, 부드러운 그림자, 올리브 포인트 색상으로 만든다.
- 사이트 favicon은 `💡` 이모지 기반 SVG 데이터 URI를 사용한다.
- 홈에서는 카드 안에 분야 라벨과 제목만 보여주되, 외부 이미지 URL 또는 `images/` 자산이 등록된 글은 카드 상단에 해당 이미지를 썸네일로 보여준다.
- 카드 썸네일과 정적 글 본문 이미지는 원본 크기를 그대로 노출하지 않고 표시 영역의 가로 폭에 맞춘 뒤 가로 2:세로 1 비율로 크롭해 보여준다.
- 모든 글의 대표 썸네일 이미지는 사이트 전체 기준으로 중복되면 안 되며, 같은 파일 경로, 같은 외부 URL, 같은 생성 결과물, 같은 원본을 리사이즈나 포맷만 바꾼 이미지를 다른 글에 재사용하지 않는다.
- 페이지 카드는 요약 문단을 표시하지 않고 콘텐츠 높이에 맞게 빈 공간 없이 보여준다.
- 화면에 표시하는 분야명은 `app.js`에서 정규화하며, `국제정치`와 `세계`는 `글로벌`, `한국 정치`는 `정치`, `문화·미디어`와 `문화미디어`는 `문화`로 보여준다.
- `글로벌` 분야 이모티콘은 지구본 `🌍`를 사용한다.
- 검색은 제목, 요약, 분야를 기준으로 동작한다.
- 카드 클릭 시 검색엔진이 바로 읽을 수 있는 `pages/<slug>.html` 정적 글 페이지로 이동한다.
- 카드 링크는 검색엔진이 읽을 수 있도록 `pages/<slug>.html` 정적 글 페이지를 우선 사용하며, 분야 링크는 `?desk=<분야>` 쿼리 라우팅을 사용한다. 기존 `?page=<slug>`, `#page`, `#desk` 주소도 `app.js`에서 호환한다.
- 정적 글 페이지는 `pages/<slug>.html`에 전체 본문과 글별 SEO 메타데이터를 포함하며, `app.js`는 같은 slug의 `pages/<slug>.md` 원본을 fetch해 상호작용용 상세 화면을 보강한다.
- `app.js`나 `style.css`처럼 모든 페이지에 영향을 주는 공용 자산을 바꾸면 브라우저 캐시 때문에 배포 직후 변경이 안 보일 수 있으므로, 같은 변경에서 `index.html`과 `pages/*.html`의 해당 자산 URL 버전 쿼리를 함께 갱신한다.
- 이미지가 꼭 필요한 글은 `pages/<slug>.html` 정적 글 페이지 안에서 외부 이미지 URL 또는 `images/` 자산을 직접 링크한다.
- 상세 본문 렌더링 시 본문 첫 줄의 `한 줄 요약:` 문장 전체와 예전 분야 표현도 `app.js`에서 제거·정규화해 보여준다.
- 상세 본문은 빈 줄 기준으로 블록을 나누며, `## 소제목` 블록만 본문 소제목으로 해석해 표시한다.
- 홈 상단에는 `최신 콘텐츠` 무한 카드 슬라이더가 있다.
- 홈 상단 최신 콘텐츠 제목에는 분야별 최신 대표글 수를 괄호 안 숫자로 함께 표시한다.
- 무한 카드 슬라이더는 카드 단위 반복 레일을 만들고, 첫 카드와 다음 반복의 첫 카드 사이 실제 렌더 거리를 계산해 마지막 카드 뒤에 첫 카드가 바로 이어지도록 동작한다.
- 홈의 분야별 섹션은 한 줄만 보여주고, 섹션별 페이지네이션과 `+ 더보기` 링크를 제공한다.
- 홈의 분야별 섹션 페이지네이션 숫자 버튼은 한 번에 최대 10개만 표시하고, 10개를 초과하면 `>`와 `<` 버튼으로 다음·이전 묶음으로 이동한다.
- 홈의 분야별 섹션 제목과 분야별 전체 보기 제목에는 해당 분야의 전체 페이지 수를 괄호 안 숫자로 함께 표시한다.
- `+ 더보기`와 헤더 분야 메뉴는 `?desk=<분야>` 라우팅으로 분야별 전체 보기 화면을 열고, 기존 `#desk=<분야>` 주소도 호환한다.
- 상세 화면에는 검색창, 음성으로 읽어주기 버튼, 같은 분야 더 보기 슬라이더, 인기 주제 슬라이더가 표시된다.
- 상세 화면과 정적 글 페이지에는 본문 소제목을 모은 목차를 본문 위에 일반 블록으로 표시하며, 목차 링크를 눌렀을 때 라우팅 재렌더링 없이 해당 소제목 위치로 이동해야 한다.
- 상세 화면과 정적 글 페이지에는 모든 글에 같은 무기명 댓글 컴포넌트를 표시하되, 댓글 데이터는 현재 글 slug 기준으로 나누어 저장한다.
- 댓글 컴포넌트는 화면 폭이 충분히 넓을 때 본문 오른쪽 여백을 활용한 사이드 패널로 표시하고, 화면 폭이 좁아지면 본문 패널 아래로 내려오게 한다.
- 상세 화면과 정적 글 페이지에서 글 제목과 음성으로 읽어주기 버튼 사이에는 요약 디스크립션이나 설명 문단을 표시하지 않는다.
- 상세 화면과 정적 글 페이지에서 작성자와 발행시각은 음성으로 읽어주기 버튼과 같은 줄에 두고 오른쪽 정렬로 표시한다.
- 정적 글 페이지에서 외부 이미지를 직접 링크하는 경우 이미지는 첫 번째 본문 소제목 바로 아래에 표시하고, 캡션에 `출처 : 도메인 주소` 형식으로 출처를 표기한다.
- manifest의 `summary`는 검색과 SEO 메타데이터용으로만 사용하고 상세 화면 본문 UI에는 표시하지 않는다.
- 음성 기본 버튼 문구는 `음성으로 읽어주기`, 재생 중 버튼 문구는 `음성읽기 중지하기`로 표시하고, 본문 로딩 중/읽는 중 상태 문구는 별도로 표시하지 않는다.

## SEO 규칙
- 기본 SEO 메타 태그는 `index.html`의 `<head>`에 둔다.
- 라우트별 제목, 설명, canonical, Open Graph, Twitter 카드, JSON-LD 구조화 데이터는 `app.js`에서 현재 화면 상태에 맞게 갱신한다.
- 홈은 `WebSite`, 분야별 전체 보기는 `CollectionPage`, 상세 글은 `Article` JSON-LD를 사용한다.
- 이미지가 등록된 정적 글은 Open Graph, Twitter 카드, `Article` JSON-LD의 `image` 메타데이터를 함께 제공한다.
- 존재하지 않는 문서는 `noindex, follow`로 표시한다.
- canonical URL은 실제 배포 주소 `https://lifeiz0415.github.io/EUREKA/`를 기준으로 만들고, 상세 글은 `pages/<slug>.html`, 분야 보기는 `?desk=<분야>`를 사용한다.
- `robots.txt`와 `sitemap.xml`은 루트에 둔다.
- `sitemap.xml`은 실제 배포 주소 `https://lifeiz0415.github.io/EUREKA/` 기준의 홈, 분야, 정적 글 페이지 URL을 담는다.
- manifest에 글을 추가하거나 slug를 바꾸면 같은 변경에서 `pages/<slug>.html`과 `sitemap.xml`도 함께 갱신한다.

## 뉴스레터 규칙
- 이 사이트의 상세 화면 상호작용은 뉴스레터 신청과 무기명 댓글 컴포넌트다.
- 뉴스레터 신청 UI는 메인 목록이 아니라 각 상세 페이지에만 노출한다.
- 뉴스레터 신청 UI는 상세 제목/설명 문구를 표시하지 않고, 이메일 입력창과 신청 버튼만 보여준다.
- 뉴스레터 이메일 입력창 placeholder는 `귀하의 이메일 주소로 최신 무료 뉴스레터를 신청하세요 ( you@email.com)`로 표시한다.
- 뉴스레터 신청 시 사용자는 관심 분야를 직접 고르지 않는다.
- 사용자가 보고 있는 페이지의 정규화된 분야가 곧 관심 분야다.
- `data.json`의 `newsletter_subscribers` 배열에 이메일과 관심 분야를 기록한다.
- 같은 이메일이 다른 분야 페이지에서 다시 신청되면 `interests` 배열에 분야를 누적한다.
- 브라우저가 File System Access API를 지원하면 연결된 `data.json` 파일에 직접 저장한다.
- 직접 저장이 불가능하면 업데이트된 `data.json` 다운로드를 생성해 사용자가 파일을 교체하도록 한다.

## 댓글 컴포넌트 규칙
- 댓글은 글별로 별도 구현하지 않고, 모든 상세 글 화면에서 재사용하는 하나의 공통 컴포넌트로 구현한다.
- 댓글 컴포넌트는 `app.js`에서 렌더링하고, `style.css`에서만 스타일링한다.
- 정적 글 페이지의 기존 본문 마크업도 `app.js`에서 공통 상세 레이아웃으로 보정해, 본문 패널과 댓글 패널이 같은 반응형 배치 규칙을 따르게 한다.
- 댓글은 무기명으로만 입력하며 이름, 비밀번호, 로그인, 회원가입, 관리자 승인, 서버 API, 데이터베이스를 도입하지 않는다.
- 댓글 저장은 브라우저 `localStorage`를 사용하고, 저장 키는 글 slug를 기준으로 나누어 같은 컴포넌트가 글별 댓글 목록을 보여주게 한다.
- 댓글은 `data.json`, `pages/*.md`, `pages/*.html`, sitemap, JSON-LD, SEO 메타데이터에 저장하거나 반영하지 않는다.
- 댓글 본문은 HTML로 해석하지 않고 반드시 이스케이프해 표시한다.
- 댓글 입력 길이는 과도하게 길어지지 않도록 제한하며, 비어 있는 댓글은 저장하지 않는다.
- 사용자가 자기 브라우저에 저장된 댓글을 지울 수 있도록 삭제 버튼을 제공한다.

- `data.json`은 최소한 아래 구조를 가진다.
- 댓글 데이터는 `data.json`에 저장하지 않는다.

```json
{
  "newsletter_subscribers": [],
  "updated_at": "ISO-8601"
}
```

- 각 구독자는 아래 구조를 따른다.

```json
  "interests": ["경제"],
  "last_page_slug": "us-china-tariff-war-2026",
  "subscribed_at": "ISO-8601",
  "updated_at": "ISO-8601"

```

## pages 폴더 규칙
- 각 페이지는 원본 `pages/<slug>.md`와 검색엔진용 생성 결과 `pages/<slug>.html` 한 쌍으로 유지한다.
- 토픽 페이지 파일명은 `분야-hot-topic-번호` 형식을 쓰지 않고, 글 제목을 영어로 요약한 kebab-case slug를 사용한다.
- slug는 제목의 핵심 명사와 독자 질문을 3~8개 영어 단어로 압축해 만든다.
- slug에는 가능하면 정규화된 분야 접두어(`technology`, `semiconductor`, `economy`, `global`, `politics`, `industry`, `science`, `culture`, `sports`)를 앞에 둔다.
- slug는 소문자 영문, 숫자, 하이픈만 사용하고 공백, 한글, 특수문자, 마침표는 넣지 않는다.
- slug 끝에는 글의 시의성을 구분할 수 있도록 필요한 경우 `2026` 같은 연도 토큰을 붙인다.
- `hot-topic`, `topic`, `issue`, 번호만 붙인 이름, 의미가 빈약한 임시 이름은 파일명으로 쓰지 않는다.
- 제목이 비슷해 slug가 겹치면 단순 번호를 먼저 붙이지 말고 제목의 차이를 드러내는 영어 핵심어를 추가해 구분한다.
- slug를 정하면 같은 slug로 `pages/<slug>.md`, `pages/<slug>.html`, `app.js`의 `topicSlugsByDesk` 또는 manifest 항목, 정적 HTML 내부 canonical/OG/JSON-LD URL, `sitemap.xml`을 모두 함께 갱신한다.
- 한 파일은 최소 3천자 이상의 장문 본문을 가진다.
- 분야와 메타데이터는 파일 안이 아니라 `app.js`의 manifest에서 관리하고, 생성된 HTML의 head 메타데이터도 이 manifest를 기준으로 맞춘다.
- 글 발행시각은 날짜만 쓰지 않고 `YYYY-MM-DDTHH:mm:ss+09:00` 형식으로 시분초와 시간대까지 기록한다.
- 글 발행시각은 임의로 생성하지 않고, 해당 글의 본문·정적 HTML·manifest·sitemap 반영이 모두 끝난 최종 생성 시점의 실제 한국시간 년월일시분초를 정확히 입력한다.
- 실제 최종 생성 시점보다 미래인 발행시각을 가진 글은 예약 글로 간주하며, 해당 시각 전에는 목록·검색·상세 화면·sitemap에서 공개하지 않고 정적 글 페이지도 `noindex, follow` 상태로 둔다.
- 새 글을 처음 생성할 때는 본문과 별개로 해당 글 내용과 직접 연결되는 대표 이미지까지 같은 작업에서 반드시 확정한다.
- 새 글 대표 이미지를 고르거나 기존 글에 썸네일을 추가할 때는 반드시 해당 글의 `pages/<slug>.md` 본문과 `pages/<slug>.html`의 첫 본문 문단, 소제목, 핵심 사례를 먼저 읽고 이미지 장면을 정한다.
- 새 글 대표 이미지는 같은 분야라는 이유만으로 공용 이미지를 재사용하지 않고, 해당 글의 제목·요약·첫 본문 문단·소제목이 가리키는 구체적 장면이나 대상에 맞는 이미지로 고른다.
- 새 글 대표 이미지는 이미 게시된 다른 글과 같은 이미지를 다시 쓰지 않는 것을 원칙으로 하며, 부득이하게 비슷한 대상을 다루더라도 다른 이미지 파일을 선택한다.
- 새 글 대표 이미지는 저작권 문제가 없는 공개 사용 가능 이미지여야 하며, 라이선스와 핫링크 가능 여부를 확인한 뒤 사용한다.
- 새 글은 대표 이미지가 `app.js` manifest의 `image` 객체, `pages/<slug>.html` 본문 이미지, Open Graph, Twitter 카드, JSON-LD `image`, 출처 캡션까지 함께 반영되기 전에는 게시 완료로 보지 않는다.
- 상세 화면과 정적 글 페이지는 글 제목 아래 도구 줄에서 음성으로 읽어주기 버튼과 같은 줄 오른쪽에 해당 분야의 순환 에이전트 이름과 발행시각을 표시한다.
- 본문은 기사문/블로그문 형식의 읽기 쉬운 한국어 존댓말 문단으로 작성한다.
- 본문은 온전히 해당 주제의 장면, 이해관계, 비용, 변화, 판단 기준만 다루며 글을 쓰는 에이전트·작성자·페르소나의 관점이나 문체를 직접 설명하지 않는다.
- `Ada는`, `Iris는`, `작성자는`, `이 에이전트는`, `문체는`, `관점에서는`, `자료의 문장을 흉내 내지 않습니다`처럼 내부 작성 방식이나 페르소나 설정을 독자에게 드러내는 문장은 본문에 쓰지 않는다.
- 에이전트별 개성은 본문 밖에서 설명하지 않고, 문장 리듬·비유·질문 순서·사례 선택의 차이로만 자연스럽게 드러낸다.
- 글은 공통 템플릿을 채운 흔적이 보이지 않게 주제별 고유 장면과 용어를 중심으로 개별 작성하며, 여러 글에 같은 문단 구조와 문장을 반복하지 않는다.
- 본문 문장 끝은 `~입니다`, `~합니다`, `~습니다`처럼 독자에게 설명하는 존댓말을 기본으로 하며, `~이다`, `~다`로 끝나는 반말·평서체 문장을 피한다.
- 한 문단이 지나치게 길어지지 않도록 빈 줄로 구분한다.
- 각 글은 첫 줄 `한 줄 요약:` 다음의 첫 본문 문단을 도입부로 두고, 그 이후 흐름을 기준으로 3~4개의 `## 소제목`을 배치한다.
- 소제목은 반복 템플릿을 쓰지 않고, 이어지는 문단 묶음의 핵심 명사·긴장·판단 기준을 압축한 고유 문구로 작성한다.
- 소제목은 본문 문장과 달리 `~입니다`, `~합니다`, `~나요` 같은 존댓말로 끝낼 필요가 없으며, `누가 비용을 감당하나`, `무엇이 출하를 막나`, `신뢰는 어디서 깨지나`처럼 짧고 자연스러운 질문형을 우선한다.
- 소제목 중복을 피하려고 slug의 영문 단어를 그대로 뒤에 붙이지 않는다. `inference`, `tools`, `fandom` 같은 영문 꼬리표 대신 글 제목에서 뽑은 자연스러운 한국어 핵심어로 구분한다.
- `왜 지금 주목받나`, `변화가 작동하는 방식`, `남은 제약과 부담`, `앞으로 확인할 기준`, `왜 지금 커졌나`, `변화가 움직이는 방식`, `부담이 남는 지점`, `앞으로 볼 기준`처럼 여러 글에 그대로 반복될 수 있는 범용 소제목은 쓰지 않는다.
- `전환점`, `작동 조건`, `남은 부담`, `확인 신호`, `검증 기준`, `확산 조건`처럼 딱딱한 고정 꼬리말을 붙여 만든 소제목을 반복하지 않는다.
- 소제목은 표지판 같은 명사구보다 독자가 다음 문단을 읽고 싶어지는 질문형·장면형 문구를 우선한다.
- 좋은 소제목은 `누가 비용을 감당하나`, `무엇이 출하를 막나`, `신뢰는 어디서 깨지나`처럼 문단 안의 갈등과 장면이 보이게 쓴다.
- 소제목은 문단 첫 문장을 그대로 복사하지 말고, 이어지는 문단 묶음에서 독자가 흥미를 느낄 핵심 질문과 판단 기준을 8~18자 안팎의 자연스러운 한국어로 요약한다.
- 한 글 안의 소제목 3~4개는 서로 다른 핵심어를 사용해 문단 전개가 보이게 만들고, 같은 분야의 다른 글과도 같은 4개 세트가 반복되지 않게 한다.
- 소제목에는 콜론식 부제 구조, 번호 나열, 과장 수식어, `결론적으로`, `종합하면`, `한마디로`, `요컨대` 같은 공식적 마무리 표지어를 쓰지 않는다.
- `pages/<slug>.md`에 소제목을 추가하면 같은 변경에서 `pages/<slug>.html`의 본문에도 같은 위치의 `<h3 class="article-subheading">` 소제목을 반영한다.
- 제목에는 AI 생성 문장처럼 보이는 콜론식 부제 구조를 쓰지 않고, 필요한 의미를 자연스러운 한국어 제목 문장으로 풀어 쓴다.
- 마지막 문단은 `결론적으로`, `결국`, `요컨대`, `정리하면`, `종합하면`, `한마디로` 같은 공식적 마무리 표지어로 시작하지 않고, 글의 실제 함의와 다음 판단 기준을 자연스럽게 남기며 끝낸다.

## 분야별 페이지 생성 플로우 규칙
- 이 사이트의 자동 토픽 페이지는 아래 플로우를 따른다.

```text
분야 확정
  ↓
분야별 최신 토픽 문장 10개 생성
  ↓
각 토픽 문장을 페이지 제목으로 사용
  ↓
`hotTopicsByDesk`에 분야별 토픽 문장 배열 등록
  ↓
`topicPages`가 각 토픽을 페이지 메타데이터로 변환
  ↓
글 제목을 영어로 요약한 slug를 `topicSlugsByDesk`에 등록하고 `pages/<slug>.md`와 `pages/<slug>.html` 파일명 결정
  ↓
각 slug에 대응하는 장문 본문을 `pages/<slug>.md`로 작성하고 `pages/<slug>.html` 정적 글 페이지를 생성
  ↓
`pages = [...featuredPages, ...topicPages]`에 합쳐져 게시 목록에 노출
  ↓
홈 카드, 분야별 섹션, 검색, 상세 페이지에서 게시 완료 상태로 표시
```

- 분야별 최신 토픽 문장은 각 분야마다 10개를 기본 단위로 만든다.
- 사용자가 여러 개의 글 작성을 요청하면 한 번에 묶어 처리하지 않고, 글 1개마다 분야 에이전트 선택, 주제 확정, slug 확정, 본문 작성, 정적 HTML 생성, manifest 반영, sitemap 반영, 검증 루틴을 순서대로 반복한다.
- 같은 분야의 글은 해당 분야 3명 에이전트가 발행 순서 기준으로 한 명씩 돌아가며 맡는다.
- 토픽 문장 하나는 페이지 하나가 되며, 카드 제목과 상세 페이지 제목의 원천이다.
- `hotTopicsByDesk`는 토픽 문장 저장소이고, 실제 화면 표시 분야는 `normalizeDesk()` 기준을 따른다.
- `topicPages`는 `hotTopicsByDesk`를 순회해 `slug`, `title`, `desk`, `publishedAt`, `summary`, 선택적 `image`를 가진 manifest 항목을 만든다.
- `topicSlugsByDesk`는 각 토픽 제목을 영어로 요약한 파일명 저장소이며, 기존 자동 토픽의 `createTopicPublishedAt(deskIndex, topicIndex)` 결과는 과거 항목 호환용으로만 취급한다. 새 글을 추가할 때는 자동 배정 시각을 그대로 쓰지 말고 최종 생성 시점의 실제 한국시간을 manifest와 정적 HTML, sitemap에 직접 반영한다.
- `topicImagesBySlug`는 자동 토픽 페이지 중 이미지가 필요한 slug와 `image` 객체 상수를 연결하는 선택 매핑이다.
- 게시가 완료되려면 manifest 항목뿐 아니라 같은 slug의 `pages/<slug>.md` 본문 원본과 `pages/<slug>.html` 정적 글 페이지가 반드시 존재해야 한다.
- 상단 `최신 콘텐츠` 슬라이더는 전체 게시 목록에서 분야별 최신 발행 페이지 한 개씩을 가져와 구성한다.

## app.js 구현 규칙
- 페이지 manifest 배열이 있어야 한다.
- 각 항목은 최소 `slug`, `title`, `desk`, `summary`를 가진다. 대표 이미지는 웹검색으로 찾은 공개 사용 가능 외부 이미지를 우선하며, 이 경우 `image.externalSrc`, `image.alt`, `image.sourceUrl`을 함께 가진다. 기존 호환을 위해 `image.src`도 사용할 수 있으나, 새 글에서는 외부 이미지 URL을 `externalSrc`에 두는 것을 우선한다. 생성형 이미지나 직접 관리 이미지 자산은 외부 이미지가 부적합할 때만 `image.generatedSrc` 또는 기존 `image.src`, `image.credit`, 필요한 경우 `image.width`, `image.height`를 함께 가진다.
- 각 항목의 `publishedAt`은 `YYYY-MM-DDTHH:mm:ss+09:00` 형식을 사용한다.
- 현재 manifest는 `featuredPages`, `hotTopicsByDesk`에서 생성하는 `topicPages`, 그리고 `pages = [...featuredPages, ...topicPages]` 구조를 사용한다.
- 제목 기반 토픽 slug 관리를 위해 `topicSlugsByDesk`를 사용하고, 자동 토픽 이미지 연결에는 `topicImagesBySlug`를 사용한다. `hotTopicsByDesk`, `deskSlugs`, `deskEmoji`에는 예전 분야 키가 남을 수 있지만 화면 표시와 뉴스레터 저장값은 `normalizeDesk()` 기준이어야 한다.
- 홈 카드 렌더링, 검색 필터, 정적 글 페이지 호환, `pages/*.md` fetch, 뉴스레터 저장은 모두 `app.js`에서 처리한다.
- 무기명 댓글 컴포넌트의 렌더링, slug별 `localStorage` 읽기·쓰기, 삭제 처리도 모두 `app.js`에서 처리한다.
- 라우트별 SEO 메타데이터 갱신도 `app.js`에서 처리한다.
- 카드 슬라이더의 실제 이동 거리 계산은 `refreshSliderLoops()`에서 처리한다.

## 이미지 규칙
- `images/` 폴더는 웹검색 이미지가 부적합할 때 쓰는 생성형 이미지 또는 직접 관리하는 공개용 이미지 자산 저장용으로 유지한다.
- 이미지가 꼭 필요한 정적 글은 먼저 웹검색으로 주제와 맞는 저작권 문제가 없는 무료 외부 이미지 URL을 찾고, 직접 생성·관리하는 이미지 파일은 외부 후보가 글 내용과 맞지 않거나 사용 조건이 불분명할 때만 `images/` 안에 저장해 상대 경로로 연결한다.
- 이미지 선정 로직의 최우선 기준은 각 글의 실제 내용과 직접 연결되는 장면성이다. 분야명, 제목 키워드, slug만으로 이미지를 배정하지 않고, 본문이 설명하는 핵심 사건·대상·행위·이해관계·비용·판단 기준을 분석한 뒤 그 글에만 어울리는 이미지를 고른다.
- 이미지 작업은 품질과 저작권 안정성을 위해 아래 `웹검색 우선 모드`를 기본 플로우로 삼는다.
- `웹검색 우선 모드`는 글 1개마다 `제목·slug 확인 → pages/<slug>.md 본문 핵심 3줄 추출 → 첫 본문 문단과 소제목 확인 → 이미지 장면 한 문장 확정 → 공개 사용 가능 외부 이미지 웹검색 → 라이선스·원출처·핫링크 가능성 확인 → app.js/정적 HTML 메타데이터 치환 → 빠른 검증 → 커밋·푸시` 순서로 처리한다.
- 생성형 이미지는 웹검색 후보가 글의 핵심 장면과 맞지 않거나, 라이선스·원출처·핫링크 가능성이 불분명하거나, 사용자가 명시적으로 생성 이미지를 원할 때만 보조 수단으로 사용한다.
- 톤 프리셋은 글의 성격에 맞춰 고른다. 인물·정치·사회 현장 글은 `사실적 다큐멘터리 사진`을 우선하고, 독자가 “AI 느낌이 없어야 한다”고 요청하면 이 프리셋을 필수로 사용한다. 기술·반도체·제품 글은 `제품·장비 클로즈업`, 추상 정책·경제 구조 글은 `현실적인 편집 사진` 또는 `절제된 정책 일러스트`, 스포츠·문화 현장 글은 `현장 스냅샷`을 우선한다.
- 사실적 이미지를 만들 때는 과장된 조명, 완벽한 대칭, 플라스틱 피부, 부자연스러운 손, 가짜 글자, 정당 로고, 숫자, 워터마크, 캠페인 구호, 특정 실존 인물과 닮은 얼굴을 피한다.
- `AI 느낌이 없어야 한다`는 요구가 있으면 공개 라이선스 실제 사진 후보를 먼저 사용하고, 글의 핵심 장면과 직접 맞는 후보가 전혀 없을 때만 사실적 사진 프롬프트로 생성한다.
- 생성 이미지는 보조 수단으로 사용할 때 기본적으로 2:1 카드 썸네일에 맞게 WebP로 저장하고, `images/<slug>.webp`처럼 slug 기반 파일명을 사용한다. 같은 이미지를 다른 글에 재사용하지 않는다.
- 한국주식·미국주식·크립토 글의 대표 이미지는 기본적으로 로고·텍스트 합성 없이 공개 사용 가능한 실제 이미지 URL을 우선한다. 사용자가 명시적으로 로고형 썸네일을 원할 때만 공식 PNG/WebP 같은 래스터 로고 자산을 보조 요소로 합성하며, AI가 로고를 임의로 그리게 두거나 SVG를 직접 그려 로고를 재현하지 않는다. 캡션과 `credit`에는 로고 출처를 함께 표기한다.
- 생성형 이미지를 쓰는 경우에만 이미지 프롬프트를 작성하며, 매번 처음부터 길게 새로 만들지 않고 선택한 톤 프리셋에 글별 핵심 장면, 대상, 장소, 피해야 할 요소만 갈아 끼워 작성한다.
- 빠른 검증은 최소 `node --check app.js`, 정적 HTML JSON-LD 파싱, `git diff --check`를 포함한다. `images/` 자산을 추가·교체한 경우에는 WebP 파일 형식 확인을 함께 수행하고, 화면 깨짐이 의심되거나 레이아웃·이미지 로딩을 바꾼 경우에는 브라우저 검증까지 추가한다.
- 여러 글을 연속 처리할 때도 본문 분석과 이미지 장면 확정은 글 1개씩 순차적으로 수행한다. 다만 검증과 커밋은 사용자가 허용하면 3~5개 단위로 묶을 수 있다.
- 외부 이미지는 저작권과 핫링크 가능성, 검색엔진 접근성을 확인한 뒤 사용한다.
- 외부 이미지는 독자가 보는 이미지 안에 텍스트, 숫자, 로고, 워터마크가 없는 후보를 우선한다.
- 외부 이미지는 글 제목이나 분야만 보고 대충 고르지 않고, 해당 글의 핵심 사건·장소·제품·행위·이해관계자를 먼저 좁힌 뒤 그 장면에 맞는 후보를 선택한다.
- 외부 이미지는 이미 다른 글에 쓴 동일 URL을 다시 쓰지 않는 것을 원칙으로 하며, 새 글마다 다른 이미지 파일을 배정한다.
- 외부 이미지 후보가 여러 개일 때는 저작권이 더 명확하고, 글 내용과 연결이 더 직접적이며, 카드 썸네일로 잘 보이는 후보를 우선한다.
- 외부 이미지의 `sourceUrl`은 가능하면 썸네일 파일 주소가 아니라 저작자, 원출처, 라이선스를 확인할 수 있는 파일 설명 페이지 또는 공식 원출처 페이지를 사용한다.
- 라이선스, 원출처, 저작자 정보가 불분명한 이미지는 주제 적합도가 높아도 사용하지 않는다.
- 이미지를 추가하면 같은 변경에서 `app.js`의 manifest 이미지 정보, 카드 썸네일, 정적 HTML 본문 이미지, 출처 캡션 또는 생성 이미지 표기, Open Graph 이미지, Twitter 이미지, JSON-LD `image`, 접근성용 `alt`를 함께 갱신한다.
- `app.js` 매니페스트에는 `thumbnail` 메타데이터를 넣지 않고, 이미지가 필요한 경우 `image` 객체만 사용한다. 웹검색 이미지는 `externalSrc`와 `sourceUrl`을 우선 사용하고, 생성 이미지나 직접 관리 이미지는 `generatedSrc` 또는 기존 호환용 `src`를 사용한다. 외부 이미지를 기본으로 쓰되 생성 이미지를 대체 후보로 남길 때는 `generatedSrc` 또는 `fallbackSrc`에 `images/` 상대 경로를 넣는다.

- 신규 에이전트의 `참고 웹사이트`는 최소 3개를 두며, 각 항목은 단순 URL 나열이 아니라 그 사이트에서 어떤 관점, 자료 구조, 검증 습관을 참고할지 한글 문장으로 설명한다.
- 참고 웹사이트를 벤치마킹할 때는 원문 표현, 제목 리듬, 고유 비유, 문장 배열을 모방하지 않고 도입 방식, 자료 확인 순서, 주장 전개, 반론 처리, 마무리 방식만 추상화해 새 한국어 템플릿으로 변환한다.
- 신규 에이전트는 벤치마킹 템플릿을 여러 개 보유해야 하며, 각 템플릿은 도입 방식, 글 구조, 설명 방식, 설득 방식, 마무리 방식을 분리해 기록한다.
- 신규 에이전트의 문장형 템플릿 수는 문단형 템플릿 수의 4배 이상이어야 한다. 현재 기본값은 문장형 32개와 문단형 8개이며, 문단형 템플릿이 늘어나면 문장형 템플릿도 비율에 맞춰 함께 늘린다.
- 신규 에이전트의 템플릿 제목, 비유 축, 질문 방식, 전개 순서, 마무리 방식은 기존 전체 에이전트 문서와 중복되면 안 된다.
- 신규 에이전트 템플릿을 생성한 뒤에는 같은 분야의 다른 두 에이전트뿐 아니라 전체 에이전트 문서와 비교해 동일하거나 거의 같은 제목·문장 구조·문단 구조가 없는지 검증한다.
- 신규 에이전트 문서가 충분한 참고 웹사이트와 템플릿을 갖추기 전에는 해당 desk의 새 글을 게시 완료로 보지 않는다.
- 같은 분야 안에서도 세 에이전트는 서로 다른 필체 개성을 가져야 하며, 문장 리듬, 설명 순서, 비유 사용, 분석 초점에서 차이를 만든다.
- 각 에이전트는 자기 문서의 참고 웹사이트에서 관점, 구성 방식, 검증 습관을 참고하되 원문 문체나 표현을 모방하지 않고 자기 페르소나의 한국어 문체로 변환한다.
- 각 에이전트 문서에는 `벤치마킹 글쓰기 템플릿` 섹션을 두고, 참고 웹사이트에서 관찰한 글 구조, 레이아웃, 주장 전개, 설명 방식, 설득 방식, 도입 방식, 마무리 방식을 문장 모방 없이 추상화해 기록한다.
- 새 글을 작성할 때는 해당 에이전트의 벤치마킹 템플릿 중 하나를 무작위 또는 순환 방식으로 고른 뒤, 주제에 맞게 변형해 사용한다.
- 벤치마킹 템플릿은 독자에게 보이는 본문에 직접 노출하지 않으며, 본문에는 `이 글은 어떤 템플릿을 따릅니다`, `참고사이트처럼`, `에이전트의 관점에서는` 같은 제작 과정 설명을 쓰지 않는다.
- 템플릿은 글의 뼈대만 제공하며, 실제 문단은 해당 주제의 고유 장면, 이해관계자, 비용, 사례, 판단 기준으로 매번 새로 작성한다.
- 각 에이전트 문서에는 실제 글을 조립할 때 활용할 수 있는 `문장형 템플릿`과 `문단형 템플릿` 섹션을 별도로 둔다.
- 문장형 템플릿은 도입 문장, 전환 문장, 대비 문장, 설득 문장, 판단 기준 문장, 마무리 문장처럼 한두 문장 단위로 재조합 가능한 표현 구조를 담는다.
- 문단형 템플릿은 도입부, 배경 설명, 이해관계자 분석, 비용·위험 분석, 반론·한계, 판단 기준, 마무리처럼 한 문단 단위로 재조합 가능한 전개 구조를 담는다.
- 문장형·문단형 템플릿은 그대로 복사해 붙이는 완성 문장이 아니라, 주제별 명사·장면·이해관계자·지표를 갈아 끼워 새 문장으로 변형하는 조립식 설계도다.
- 각 에이전트는 자기 페르소나만의 중독성 있는 리듬, 질문 방식, 비유 방식, 설득 순서, 마무리 여운을 가진 템플릿을 다수 보유해야 하며, 템플릿 수와 다양성이 곧 페르소나 품질의 핵심이다.
- 서로 다른 에이전트의 문장형·문단형 템플릿은 제목, 문장 구조, 비유, 전개 순서, 마무리 방식이 중복되면 안 된다.
- 새 템플릿을 추가할 때는 같은 분야 안의 다른 두 에이전트뿐 아니라 전체 36개 에이전트 문서와 비교해 동일하거나 거의 같은 문장 구조가 없는지 확인한다.
- 글 작성 시 에이전트는 자기 문서의 문장형·문단형 템플릿을 여러 개 조합하되, 최종 공개 본문에는 템플릿명·페르소나명·참고사이트·작성 방식이 드러나지 않게 자연스러운 장문으로 녹인다.
- 새 글 작성 시 같은 분야의 다른 에이전트와 문장 리듬, 설명 순서, 비유 사용, 분석 초점이 겹치지 않도록 차별점을 먼저 정한다.
- 필체 개성은 `AGENTS.md`의 제목·본문·소제목·SEO 규칙보다 우선하지 않는다.
- `app.js`의 작성자 표시와 정적 HTML의 `article-meta`, JSON-LD `author.name`은 같은 순환 규칙을 따른다.
- 파일명은 kebab-case를 유지한다.

## 개발 원칙
- 새 기능을 추가할 때 먼저 이 구조를 깨지 않는지 확인한다.
- 단일 정적 앱이라는 현재 제약을 우선한다.
- 사용자가 구조 축소를 원하면 새 디렉터리를 늘리지 않는다.
- 파일 수를 늘리기보다 현재 파일 안에서 해결하는 쪽을 우선한다.
- 코드 변경이 있는 작업은 완료 전에 `AGENTS.md`를 실제 파일 구조와 실제 코드에 맞게 갱신했는지 확인한다.
- 코드 변경이 있는 작업을 완료하면 변경사항을 의미 있는 단위로 한글 커밋 메시지를 사용해 git commit하고 GitHub 원격 저장소에 push한다. 단, 사용자가 해당 작업에서 커밋 또는 푸시 금지를 명시한 경우에는 그 지시를 우선한다.

## 완료 기준
- 루트에는 `AGENTS.md`, `README.md`, `index.html`, `style.css`, `app.js`, `data.json`, `robots.txt`, `sitemap.xml`, `pages/`, `agents/`, `images/`만 남아 있어야 하며, 글별 HTML은 `pages/` 안에 둔다.
- 홈 카드와 상세 보기, 상세 페이지 뉴스레터 신청, 무기명 댓글 작성·삭제가 동작해야 한다.
- `pages/` 안의 각 문서는 장문이어야 한다.
- 문서와 실제 코드가 서로 충돌하지 않아야 한다.
